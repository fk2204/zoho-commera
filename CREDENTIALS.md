# CREDENTIALS.md — Where Secrets Live & How to Obtain Them

**CRITICAL: This file documents WHERE credentials live, NOT the values themselves. Never commit actual secrets to this repository.**

---

## What Lives Where

| Credential | Location | File Permissions | Scope | Notes |
|-----------|----------|-----------------|-------|-------|
| **Client ID** | `.env` | Gitignored | Not sensitive (public client identifier) | Obtained from Zoho API Console |
| **Client Secret** | `.env` | Gitignored | SENSITIVE | Never log, never share, rotate if exposed |
| **Refresh Token** | `~/.zoho/tokens.json` | Mode 0600 (outside repo) | SENSITIVE | Grants all authorized scopes, long-lived |
| **Access Token** | In-memory only | RAM only | SENSITIVE | Short-lived (auto-refreshed), never persisted |
| **SMTP User** | `.env` | Gitignored | SENSITIVE | Email sender account |
| **SMTP Password** | `.env` | Gitignored | SENSITIVE | App-specific password from Zoho Mail |
| **Org IDs** | `.env` | Gitignored | Not sensitive (organization identifiers) | Required for Books, Inventory, Desk, Projects |

## Required Variables & Where to Obtain Them

### 1. **ZOHO_CLIENT_ID** (Self Client ID)

**What it is:** Public identifier for your OAuth application  
**Where to get it:**
1. Go to https://api-console.zoho.com
2. Log in with your Zoho account
3. Select your organization
4. Click **"Self Client"** (not OAuth Clients)
5. Copy the **Client ID** field
6. Paste into `.env`

**Example:**
```
ZOHO_CLIENT_ID=1000.xxxxxxxxxxxxx
```

**Scope:** Used for all API calls; safe to reference in logs with redaction context  
**Rotation:** Only if exposed; regenerate in API Console  
**Risk if leaked:** Attackers need Client Secret + Refresh Token to be dangerous

---

### 2. **ZOHO_CLIENT_SECRET** (Self Client Secret)

**What it is:** Private key that proves your identity to Zoho  
**Where to get it:**
1. Go to https://api-console.zoho.com
2. Click **"Self Client"**
3. Copy the **Client Secret** field (only shown once; cannot be retrieved later)
4. Paste into `.env`
5. **SAVE THIS IMMEDIATELY** — if you leave the page, you must regenerate

**Example:**
```
ZOHO_CLIENT_SECRET=xxxxxxxxxxxxxxxxxxxxx
```

**Scope:** SENSITIVE — combined with Client ID, allows token refresh  
**Rotation:** Immediately if lost or exposed; regenerate in API Console, update `.env`  
**Risk if leaked:** Attackers can refresh your access tokens indefinitely

**NEVER:**
- Log this value (logger redacts it)
- Commit to git (`.gitignore` blocks `.env`)
- Paste in chat, email, or screenshots
- Use in URLs or query parameters

---

### 3. **Refresh Token** (OAuth Grant Token)

**What it is:** Long-lived token that generates short-lived access tokens  
**Where it comes from:**
- Generated during initial OAuth setup: `npm run setup:oauth`
- Stored automatically in `~/.zoho/tokens.json`

**Example location:**
```
~/.zoho/tokens.json (macOS/Linux)
C:\Users\[YourUsername]\.zoho\tokens.json (Windows)
```

**File structure:**
```json
{
  "refreshToken": "1000.xxxxxxxxxxxxxxxxxxxxx",
  "expiresAt": 1735689600000,
  "scope": "CRM.Leads.ALL,CRM.Contacts.ALL,..."
}
```

**File permissions (CRITICAL):**
```bash
# Verify permissions are 0600 (read/write owner only)
ls -l ~/.zoho/tokens.json
-rw------- 1 user user 1234 Apr 27 12:00 /home/user/.zoho/tokens.json ✅

# If wrong, fix immediately:
chmod 0600 ~/.zoho/tokens.json
```

**Scope:** SENSITIVE — grants ALL authorized apps and modules  
**Rotation:** If exposed, revoke at https://accounts.zoho.com/home#sessions/userauthtoken  
**Risk if leaked:** Attackers have full access to all 11 Zoho apps with your permissions

**NEVER:**
- Store in `.env` (too easy to commit)
- Share the file
- Copy to logs or error messages
- Leave file world-readable

---

### 4. **Access Token** (Session Token)

**What it is:** Short-lived token (1 hour) that proves you're authenticated  
**Where it lives:** In-memory only, in the Node.js process  
**Auto-generated by:** `src/auth/oauth.js` using Client ID + Secret + Refresh Token  
**Lifespan:** 1 hour, then auto-refreshed

**You don't need to:**
- Store this anywhere
- Obtain it manually
- Manage its lifecycle

The refresh logic is automatic:
- First API call → refresh token expires soon → mint new access token
- All subsequent calls use the new token
- Single-flight mutex prevents simultaneous refresh races

### 5. **ZOHO_SMTP_USER** (Mail Account)

**What it is:** Email sender address for transactional emails  
**Where to get it:**
1. Log into Zoho Mail: https://mail.zoho.com
2. Go to **Settings** → **Accounts**
3. Look for your primary email address (e.g., `applications@commerafunding.com`)
4. Copy the full email address
5. Paste into `.env`

**Example:**
```
ZOHO_SMTP_USER=applications@commerafunding.com
```

**Scope:** Used by email automation jobs (send-application-confirmation, etc.)  
**Risk if leaked:** Attackers can send emails from your account (phishing vector)

---

### 6. **ZOHO_SMTP_PASS** (Mail App Password)

**What it is:** App-specific password for SMTP authentication (NOT your main Zoho password)  
**Why app password?** Zoho doesn't allow main password over SMTP; app passwords are revocable without changing main password

**Where to get it:**
1. Log into Zoho Mail: https://mail.zoho.com
2. Go to **Settings** → **Connected Accounts** (or **Account Security**)
3. Click **"Create App Password"** (or **"Generate"**)
4. Enter app name: `zoho-commera-smtp`
5. Zoho shows a one-time password (16-32 chars)
6. Copy the password
7. Paste into `.env` IMMEDIATELY (cannot be retrieved later)

**Example:**
```
ZOHO_SMTP_PASS=xxxxxxxxxxxxxxxx
```

**Scope:** SENSITIVE — allows sending emails from your account  
**Rotation:** Revoke in Zoho Mail settings if exposed; regenerate and update `.env`  
**Risk if leaked:** Attackers can send emails from your account

**NEVER:**
- Use your main Zoho password here
- Log this value
- Commit to git

### 11. **ZOHO_TOKEN_STORE_PATH** (Optional)

**What it is:** Custom path to store the refresh token  
**Default:** `~/.zoho/tokens.json`  
**Example override:**
```
ZOHO_TOKEN_STORE_PATH=/secure/vault/zoho-tokens.json
```

**Use case:** Move tokens to a central secret manager (AWS Secrets Manager, HashiCorp Vault, etc.)

**When to set:**
- Only if you're replacing the file-based token store with an external secret manager
- For production deployments

**For local development:** Leave unset (uses default `~/.zoho/tokens.json`)

## OAuth Token Setup (First Time)

**One-time setup required. After this, tokens auto-refresh.**

### Prerequisites
1. Zoho account with API access
2. Self Client registered at https://api-console.zoho.com
3. Client ID and Client Secret saved

### Setup Commands

**Default scope (CORE tier — CRM, Mail, WorkDrive, Cliq, Sign):**
```bash
npm run setup:oauth
```

**Minimal scope (CRM + profile only):**
```bash
npm run setup:oauth -- --minimal
```

**Full scope (all 11 apps):**
```bash
npm run setup:oauth -- --full
```

### What Happens
1. Opens a browser to `https://accounts.zoho.com/oauth/v2/auth?...`
2. You log in and approve requested scopes
3. Zoho redirects back to localhost with an authorization code
4. Script exchanges code for refresh token
5. Refresh token saved to `~/.zoho/tokens.json` (mode 0600)
6. Confirmation logged

### If Setup Fails
- **"Invalid client ID"** → Check ZOHO_CLIENT_ID in `.env` matches API Console
- **"Invalid secret"** → Check ZOHO_CLIENT_SECRET matches API Console (regenerate if lost)
- **"Consent not granted"** → You declined scopes; re-run setup
- **"Scope mismatch"** → Your Zoho plan doesn't include a requested app; use `--minimal` instead

## Security Rules

### DO:
- ✅ Store Client ID/Secret in `.env` (gitignored)
- ✅ Store Refresh Token in `~/.zoho/tokens.json` (mode 0600, outside repo)
- ✅ Let Access Tokens live in RAM only
- ✅ Rotate credentials if exposed
- ✅ Verify file permissions: `ls -l ~/.zoho/tokens.json`
- ✅ Use different Org IDs per environment (dev, staging, prod)
- ✅ Revoke tokens at https://accounts.zoho.com/home#sessions/userauthtoken if leaked

### DON'T:
- ❌ Commit `.env` to git (blocked by `.gitignore`)
- ❌ Log sensitive values (logger redacts them)
- ❌ Store secrets in code (use env vars only)
- ❌ Share `.zoho/tokens.json` with teammates
- ❌ Use main Zoho password for SMTP app password
- ❌ Make `.zoho/tokens.json` world-readable
- ❌ Paste secrets in chat, email, or screenshots
- ❌ Use Client Secret in client-side code (e.g., browsers)

## Environment-Specific Setup

### Local Development
- `.env` → gitignored, contains client credentials
- `~/.zoho/tokens.json` → user home directory (mode 0600)
- No secrets committed to git
- Manual setup: `npm run setup:oauth`

### Production Deployment
- Client ID/Secret from CI/CD secret manager (not `.env`)
- Refresh Token from AWS Secrets Manager / Vault (not file-based)
- Replace `src/auth/tokenStore.js` with secret manager backend
- Different org IDs per environment (prod app IDs differ from dev)
- No manual setup; secrets injected at deploy time

### CI/CD Pipeline
- Never commit `.env`
- Inject secrets at build time via CI/CD platform (GitHub Actions, GitLab CI, etc.)
- Use secret masking to redact secrets from logs
- Test with `DRY_RUN=true` before committing changes

## Debugging & Verification

### Check what's currently configured:
```bash
# Show .env keys (not values):
grep "^ZOHO_" .env | cut -d= -f1

# Show which apps have tokens:
npm run test:connection
```

### Verify token file permissions:
```bash
ls -l ~/.zoho/tokens.json
# Should show: -rw------- (mode 0600)
```

### Force refresh access token (if auth fails):
```bash
npm run refresh-token
```

### View what scopes are in the current token:
```bash
node -e "console.log(require('./src/auth/tokenStore.js').read().scope)"
```

## Rotation & Revocation

### If Client Secret is exposed:

1. Go to https://api-console.zoho.com
2. Click **"Self Client"** → **"Regenerate Secret"**
3. Update `.env` with new secret
4. Old refresh tokens still work (until revoked below)
5. Optionally revoke existing refresh tokens:
   - Go to https://accounts.zoho.com/home#sessions/userauthtoken
   - Click **"Revoke"** on the Zoho Commera session
   - Re-run `npm run setup:oauth` to get a new refresh token

### If Refresh Token is exposed:

1. Go to https://accounts.zoho.com/home#sessions/userauthtoken
2. Find the Zoho Commera session
3. Click **"Revoke"** (immediately invalidates all access tokens from this refresh token)
4. Delete `~/.zoho/tokens.json`
5. Re-run `npm run setup:oauth` to generate a new refresh token

### If SMTP password is exposed:

1. Go to Zoho Mail → **Settings** → **Connected Accounts**
2. Find the app password for `zoho-commera-smtp`
3. Click **"Revoke"** or **"Delete"**
4. Generate a new app password (see section 6 above)
5. Update `.env` with new password
